## 2026-05-21 — html-comment-skip-markdown-aware — Parser HTML-comment-skip is paused inside fenced code blocks and transcript blocks

**Decided:** The parser's HTML-comment-skip rule (defined in `format-spec.md`, anchored in V1-D7) applies only outside two protected Markdown contexts: (a) fenced code blocks (` ``` ` … ` ``` `, including language-tagged variants); (b) `[DECISION_TRANSCRIPT]` … `[/DECISION_TRANSCRIPT]` blocks. Inside those spans, literal `<!--` / `-->` sequences are preserved byte-for-byte as content.

**Rejected:**
- global pre-tokenization string strip of `<!--...-->` — silently destroys any HTML-comment content inside transcripts (LLM exchanges may contain HTML-comment markers) and inside fenced code blocks (documentation showing literal HTML); exactly the silent-corruption seam Lesson 21 Type B captures
- author-discipline ("escape HTML comments inside transcripts") — fails the moment an LLM authors a transcript naïvely; not the structural form Lesson 2 requires

**Mechanisms:** test:html-comment-skip-outside-fenced, test:html-comment-preserved-inside-fenced, test:html-comment-preserved-inside-transcript

**Invalidates if:** Markdown dialect changes such that fenced-block delimiters become ambiguous; `[DECISION_TRANSCRIPT]` block syntax is replaced

**Context:** Surfaced by Gemini's Lesson 21 seam audit during the 2026-05-21 review. V1-D7's original HTML-comment-skip pin was load-bearing for the `mitos init` seeded-sample-wrap, but stated without the Markdown-context guard would have silently stripped any literal `<!-- ... -->` inside a pasted LLM transcript — destroying provenance the canonical core depends on. The pause-inside-fenced-and-transcript rule is the structural form.



## 2026-05-21 — signals-insert-or-ignore — Writes against signals partial unique indexes use INSERT OR IGNORE semantics

**Decided:** All writes against the `signals` table's partial unique indexes (currently `drifted` singleton-per-node, `source_reencounter` first-encounter-per-actor) use `INSERT OR IGNORE` / `ON CONFLICT(...) DO NOTHING` semantics. A duplicate signal-row write is a silent no-op at the DB engine; it does NOT raise `UNIQUE constraint failed` and cannot abort the parent transaction. The rule covers v0.2's Drift consumer writing to the `drifted` slot.

**Rejected:**
- bare INSERT (raise on duplicate) — under V1-D10 atomicity the constraint error aborts the entire entry commit; an entry that triggers a `source_reencounter` signal on first sync would persistently fail to sync on every subsequent run, because the second attempt re-tries the same INSERT
- application-layer pre-check ("query if signal exists, then INSERT") — race-condition-prone and adds two round-trips per commit when the structural form costs nothing

**Mechanisms:** test:signals-duplicate-source-reencounter-silent-noop, test:entry-resync-survives-after-first-source-reencounter, candidate-invariant-mi-4

**Invalidates if:** the partial-unique-index strategy is replaced by a different dedup mechanism (e.g., explicit signal versioning); SQLite drops `ON CONFLICT DO NOTHING` support (extremely unlikely)

**Context:** Surfaced by Gemini's Lesson 21 seam audit + Lesson 22 (failure-taxonomy for unique-constraint writes) during the 2026-05-21 review. The MI-4 partial unique index on `source_reencounter` is correct in isolation; V1-D10's atomicity contract is correct in isolation; together without the INSERT OR IGNORE pin they create a persistent-sync-failure trap exactly matching Lesson 22's "per-unit failure that becomes a cascade" pattern. Pinned at V1-D10 with cross-reference from §8.2.



## 2026-05-21 — commit-delta-cascade-transitive — CommitDelta cascade includes second-order V1-D18 dependency consequences

**Decided:** `CommitDelta.cascade_affected_scopes` (V1-D22) is computed first-order AND transitive. When a commit's edge writes flip a decision's active-status (e.g., new Decision B supersedes Decision A), the commit helper MUST also evaluate the V1-D18 stage-2 resolution dependency: every `open_question` that A `resolves` flips `resolved → parked` as a *consequence* of A leaving active, and those OQs' scopes are included in `cascade_affected_scopes`.

**Rejected:**
- first-order-only cascade (evaluate only direct edge targets) — an OQ resolved by a now-superseded decision would silently zombie-render as "resolved" in the per-scope axiom file even though its computed state is `parked`; the seam is invisible at first-order and only surfaces under the V1-D18 dependency, exactly the Lesson 21 Type A mechanism-interaction trap
- defensive re-render of all scopes on any decision-status flip — sidesteps the correctness problem at the cost of V4's per-scope render efficiency

**Mechanisms:** test:commit-delta-cascade-transitive-oq-resolution, candidate-invariant-mi-6

**Invalidates if:** V1-D18's two-stage OQ state computation is replaced by a different dependency model; a future v0.2 edge type introduces a third-order dependency chain that the one-step walk doesn't cover (would force generalization to fixed-point evaluation, not a v0.1 concern)

**Context:** Surfaced by Gemini's Lesson 21 seam audit during the 2026-05-21 review. V1-D18 (OQ self-healing) and V1-D22 (cascade) are individually correct but compose into a silent-zombie-render seam without the transitive rule. The fix walks one step through the V1-D18 dependency for any decision whose active-status flips — sufficient for V1's edge catalog and naturally extensible.



## 2026-05-21 — edge-target-resolution-case-insensitive — Slug→ID lookup at edge resolution is case-insensitive with ambiguity backstop

**Decided:** Slug→ID lookup at edge resolution is case-insensitive (`Supersedes: [Use-SQLite]` matches `use-sqlite`), even though `node_slug_norm` preserves authored casing in `nodes.slug`. If the case-insensitive lookup returns multiple active matches (case-only variants legitimately coexisting), the commit fails with a structured ambiguity error listing candidates so the user can rewrite the citation with exact casing.

**Rejected:**
- case-sensitive lookup — hard-fails otherwise-valid LLM authoring on cosmetic casing churn; LLMs author edge citations fuzzily across sessions
- silent first-match tiebreaker on ambiguous case-only collisions — silently corrupts authorial intent at the rare structural edge

**Mechanisms:** test:edge-resolution-case-insensitive, test:edge-resolution-ambiguity-error, lint:edges-fk-on-nodes

**Invalidates if:** node slugs become canonically case-folded (would make this rule redundant); a multi-byte case-folding consideration emerges that the simple ASCII-fold can't honor

**Context:** Lifted out of V1-D6 (Dangling Edge Contract) during the 2026-05-21 review with Gemini. V1-D6 had grown into the longest §4 ADR; the case-insensitive sub-rule is the volatile LLM-UX defense most likely cited independently from review rounds, so it earned its own ID (V1-D23) and citation handle.



## 2026-05-21 — commit-delta-cascade-contract — Every per-entry commit returns a structured CommitDelta with exhaustive cascade scopes

**Decided:** Every per-entry commit transaction returns a structured `CommitDelta` payload to V3b containing `node_id`, `node_scope`, `self_old_scope`, `commentary_fields_changed`, and `cascade_affected_scopes` — the last being the scope of every target node whose computed active/parked/resolved view-membership flipped due to this commit's edge writes. V4 unions `cascade_affected_scopes` with `{node_scope, self_old_scope}` into the scope-rerender set.

**Rejected:**
- defensive re-render of all scopes every sync — wasteful at corpus scale; doesn't scale with V4's per-scope render fan-out
- V4 polling active views to discover what changed — V4 can detect what is active now but not whose membership flipped this commit; without the delta, edge-driven cascade flips are invisible
- emit a separate event per affected node — multiple per-commit emissions break the per-entry atomic-transaction story

**Mechanisms:** test:commit-delta-cascade-cross-scope-kill-edge, test:commit-delta-cascade-no-edges-empty, candidate-invariant-mi-6

**Invalidates if:** v0.2+ introduces an edge type whose write changes active-view membership but isn't covered by the structural "did the target's view membership flip during this transaction?" predicate; V4's per-scope render model is replaced by a different render strategy

**Context:** Lifted out of V1-D16 (Commentary-update semantics) during the 2026-05-21 review. V1-D16 had grown into a 5-mechanism Russian doll; `CommitDelta` is independently cited from §5.1 C3, §5.2, and MI-6, so it earned its own ADR (V1-D22) and the cascade-exhaustiveness contract is now the unambiguous V1↔V3b/V4 render-targeting seam.



## 2026-05-21 — declarative-mirror-edge-reconciliation — Outgoing edges declaratively mirror the buffer, not write-once-preserve

**Decided:** On every commit (new-node insert OR hash-match commentary update), the store declaratively reconciles the node's outgoing edges to match the `ParsedEntry` exactly: edges declared in the entry are upserted; existing outgoing edges originating from this node that are no longer declared are DELETEd. Reconciliation runs inside the per-entry atomic transaction. Edges are mutable post-commit because they are not part of the canonical-core hash.

**Rejected:**
- write-once-preserve (the transcripts model) — phantom edges from past hallucinations or typos would silently corrupt every downstream consumer; a user who typos `Supersedes: [wrng-slug]` must be able to fix the line in the buffer and have the graph follow
- additive-only (never DELETE) — same failure mode plus no way to take back a supersession
- persisted retired-sticky state on the target side — overrides M3 state-from-edges; fights compute-from-edges discipline

**Mechanisms:** test:edge-reconciliation-on-commentary-update, test:edge-only-reparse-triggers-reconciliation, test:target-resurrection-on-supersedes-removal, candidate-invariant-mi-5

**Invalidates if:** edges become part of the canonical-core hash (would make them immutable and force a different reversibility model); a v0.2+ edge-audit-trail requirement makes the DELETE path lossy in an unacceptable way

**Context:** Lifted out of V1-D16 during the 2026-05-21 review. The reversibility this enables — target resurrection on kill-edge removal — is the same M3-compute-from-edges discipline that powers V1-D18's OQ self-healing resolution. Pinning declarative-mirror as its own ADR (V1-D21) clarifies that the reconciliation is design intent, not a bug worth "fixing" by persisting retired-sticky state. The Target Resurrection paragraph moved into V1-D11 (Computed State) in the same amendment since it is a corollary of M3 + declarative-mirror, not its own rule.
